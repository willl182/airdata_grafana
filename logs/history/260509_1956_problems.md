# Problem Record (Troubleshooting)

**Severity**: Medium

## The Problem

The V2 plan had grown into a single large technical document. That made it harder to distinguish:

- the master roadmap,
- what the agent should implement,
- what the user needs to decide or do manually,
- which tasks are serial vs parallel.

## User-Requested Corrections

The user requested a different organization:

- one master plan for the current stage,
- one subplan for agent-executed work,
- one subplan for user responsibilities.

## The Solution / Workaround

Created three separate planning documents:

- `plan_maestro_v2.md`: master plan, architecture, phases, critical path and parallelization.
- `subplan_agente_v2.md`: implementation tasks for the agent with closure criteria.
- `subplan_usuario_v2.md`: user decisions, VPS preparation, validation, deployment participation, and operational policies.

Updated `plan_v2.md` to clarify that it is now the extended technical reference rather than the execution tracker.

## Learnings to Preserve

- For multi-party work, do not keep all planning in one document.
- Separate architecture/reference from execution plans.
- Explicitly split agent work from user work when external access, VPS setup, DNS, or manual validation are involved.
- Keep closure criteria per phase so progress can be checked without rereading the full architecture document.
