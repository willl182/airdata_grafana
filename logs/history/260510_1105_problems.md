# Problem Record (Troubleshooting)

**Severity**: High

## The Problem

Un job creado desde la web terminaba como `completed`, pero la UI mostraba 0 lotes, sin panel, sin rango y sin artefactos. En logs del contenedor se veia que la descarga y el CSV final se generaban, pero las rutas eran globales:

```txt
data/raw/...
data/csv/final_long.csv
```

La API y la web esperaban artefactos por job en:

```txt
data/jobs/<jobId>/
```

## User-Requested Corrections

- El usuario reporto que no habia boton de cancelar y luego mostro captura donde el job aparecia `COMPLETADO` con 0 lotes.
- El comportamiento esperado es que la UI vea lotes, logs y artefactos del job.

## The Solution / Workaround

El problema estaba en `normalizeJob()` dentro de `src/grafana/jobs.ts`. Como `DEFAULT_CONFIG.outDir` era `"data"`, los jobs creados desde la web heredaban ese valor aunque no hubieran definido `outDir`.

Se cambio el default de jobs a:

```ts
const outDir = rawJob.outDir || process.env.OUT_DIR || path.join("data", "jobs", id);
```

Esto conserva `outDir` explicito y `OUT_DIR`, pero para jobs HTTP/web usa `data/jobs/<jobId>` por defecto.

## Learnings to Preserve

- Para jobs web/API, no usar `DEFAULT_CONFIG.outDir = "data"` como salida final del job.
- Si la UI muestra `completed` con 0 chunks, revisar si `job.json`, `chunks.jsonl` y `artifacts.json` existen dentro de `data/jobs/<jobId>`.
- Los jobs creados antes del fix pueden quedar inconsistentes; usar un nuevo ID despues de reconstruir Docker.
- Despues de cambios en TypeScript usados por Docker, reconstruir con `docker compose down && docker compose up --build`.

## Key Files Affected

- `src/grafana/jobs.ts`
- `src/api/server.ts`
- `webapp/src/app/_components/job-detail.tsx`
- `README.md`
